## 场景 2 · 技术架构与整体分层（01:00–02:00）

**画面**：左侧展示工程目录树（`recruit/` 后端 + `frontend/` 前端），右侧画一张分层架构图（数据源 → 数仓 ADS → Spring Boot 接口 → Vue3 大屏 / AI 浮窗）。

**旁白**：
> 系统采用前后端分离架构。后端是 **Spring Boot 4.1.0 + Java 21**，用 `JdbcTemplate` 直连本机 MySQL 的 `bigdata` 库；前端是 **Vue3 + Vite + ECharts**，深色科幻大屏风格。中间通过一套**配置驱动的 RESTful 接口**衔接——这正是本系统的核心设计思想：**18 个指标，共用一套接口，零重复代码**。AI 问答模块则接入了 DeepSeek 大模型。下面我们先看数据从哪来。

**镜头**：架构图逐层高亮（数据层→接口层→展示层）。

## 场景 3 · 数据源与数仓链路（02:00–03:10）

**画面**：展示 `智联招聘数据库2025.csv` 字段片段（19 列），然后切换到一个处理流程图：CSV → Hive 清洗(DWD) → Spark 分析 → ADS 结果表（MySQL）。打开 `sql/` 目录，点开 `generate_ads_geo_demand.py` 等生成脚本。

**旁白**：
> 原始数据是一份 19 列的 CSV。我们先用 **Hive/Spark** 做清洗与聚合，产出标准的 ADS 结果表，再同步到 MySQL 的 `bigdata` 库——一共 18 张 `ads_*` 结果表。这里强调一个工程约定：**MySQL 结果表就是分析层与前端层之间唯一的"集成契约"**，前端不关心数仓怎么算，只认接口返回的数据。每张结果表命名形如 `ads_<板块>_<指标>`，接口路径则对应 `/api/<板块>/<指标>`，天然对齐。

**镜头**：grep 一下 `ads_` 表名快速过场，强调"命名即契约"。

## 场景 4 · 后端接口设计：配置驱动（03:10–05:10）

**画面**：打开 `src/main/java/.../config/MetricRegistry.java`，镜头停在 `List.of(...)` 那段；再打开 `MetricController.java` 的 `data()` 方法。

**旁白**（慢讲，配合代码高亮）：
> 这是整个后端最巧妙的地方——**配置驱动**。大家看 `MetricRegistry`，所有 19 个指标就是一行行 `MetricDef` 记录：它声明了板块、指标名、中文标题、对应的 ADS 表名、图表类型、字段映射、排序与限制。比如这一条：`geo/demand` 指向 `ads_city_demand` 表，横向柱状图，按 `job_count DESC` 排序，并且 **limit=0 表示返回全量 321 城**——专门给大屏 KPI 用。
>
> 再看 `MetricController`，它只有一个通用方法 `GET /api/{board}/{metric}`。它拿到路径里的板块和指标，**不是去拼 SQL**，而是去注册表里查定义，再按定义组装 `SELECT * FROM 表 ORDER BY ... LIMIT ...`。路径变量只用来查表，**根本不参与 SQL 拼接，所以从根上杜绝了注入**。
>
> 所有接口统一返回这个信封：`{code, msg, data}`，`code=0` 成功。前端只要认这个格式。

**镜头**：把 `MetricController.data()` 的 SQL 拼装片段放大朗读；再 `curl` 演示一次 `GET /api/talent/edu` 返回真实 JSON。

**旁白**：
> 这样做的好处是：以后要加一个新指标，比如"远程岗位分布"，**只需要在注册表里加一行，不需要新增任何 Controller，前端也会自动渲染**。可扩展性直接拉满。

## 场景 5 · 指标注册表与代码走读（05:10–06:20）

**画面**：列出指标总表（可截图 `doc/接口文档.md` 第 5 章），按五大板块分组：人才 6 个、地域 6 个、行业 3 个、时间 1 个、文本 3 个（含 1 个内部地图数据源）。再打开 `MetricDef.java`（record）说明字段，`MetricMeta.java` 说明"对外只暴露展示所需字段、表名等内部字段不下发"。

**旁白**：
> 我们一共注册了 19 个指标，覆盖五大板块。注意一个细节：`MetricDef` 内部有 `tableName / orderBy / limit / groupBy` 这些字段，但它们**不会**出现在下发给前端的 `MetricMeta` 里——前端只需要知道"怎么画"，不需要知道"数据从哪张表来"。这种**内外分离**，既保证了前端简单，又守住了数据安全边界。此外，像学历热力图这类需要固定坐标轴顺序的，我们在注册表里直接写死 `xOrder / yOrder`，避免前端各画各的。

**镜头**：快速滚一遍指标表，重点停留 `talent/salary` 热力图与 `geo/demand` 全量两项。

## 场景 6 · 前端可视化大屏（06:20–08:00）

**画面**：打开 `frontend/src/`，展示 `views/Dashboard.vue`（统一网格布局，地图 2×2、其余 1×1）、`components/MetricChart.vue`（通用图表组件）、`components/ChinaMap.vue`（省级地图）。

**旁白**：
> 前端是一块深色科幻风大屏。核心是 `Dashboard.vue` 用 CSS Grid 做了**统一网格**：中间地图占 2×2，其余图表各占 1×1，视觉整齐。所有图表都由**一个通用组件 `MetricChart.vue`** 渲染——它根据后端下发的 `chartType` 自动切换柱状、饼图、折线、热力、堆叠、词云六种形态，真正做到了"配置驱动前端"。
>
> 中间的地图我们走了**写死本地资源**的稳妥方案：省级边界 `china-provinces.json` 和省级岗位数 `province_jobs.json` 都固化在 `public/geo` 下，前端只读本地、不依赖外网。这里有个细节设计——`visualMap` 的 `min` 设成 50，让**岗位少于 50 和等于 0 的省份同色**，避免稀疏数据误导；同时**台湾省用金色描边 + 常驻标注重点显示**。整体风格统一，玻璃拟态 + 霓虹扫描线，和主题浑然一体。

**镜头**：浏览器全屏展示大屏，鼠标划过几个图表；放大地图说明台湾省高亮与同色规则。

## 场景 7 · AI 智能问答模块（08:00–09:00）

**画面**：打开 `AiController.java`，高亮三个写死常量（URL / API Key / model=`deepseek-v4-pro`）与 `chat()` 方法；再展示右下角 `ChatWidget.vue` 浮窗。

**旁白**：
> 最后是 AI 模块。我们在右下角做了一个**可拖拽的浮窗**，点开就能和大模型对话。后端 `AiController` 接入的是 **DeepSeek `deepseek-v4-pro`**。这里有两个工程亮点：
> 第一，**数据准确性优先**——系统提示词里直接写死了平台的核心真实统计（学历分布、TOP 城市、行业薪资、技能词云……），并约束模型"只能依据真实数据作答、不得编造"，所以回答可信、可溯源。
> 第二，为了绕开受控环境无法下载 Maven 依赖的问题，整个调用用**纯 JDK 实现**：手工拼 JSON、手工提取 `content` 字段，不引入任何 JSON 库，后端零额外依赖就能跑。浮窗交互上，我们按需求做成**仅 ESC 关闭**，简洁不干扰。

**镜头**：点开浮窗，输入"北京岗位多还是上海多？"，展示 AI 基于真实数据回答；切回代码高亮 `deepseek-v4-pro` 常量与 `extractContent()`。
