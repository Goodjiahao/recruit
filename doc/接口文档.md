## 1. 概述

平台采用**配置驱动**的通用数据接口：所有业务指标共用一套 RESTful 接口，指标与所对应 ADS 结果表的映射集中在后端 `MetricRegistry` 中维护。前端大屏（Vue3 + ECharts）仅根据后端下发的"元信息"渲染图表，无需为每个指标写专用请求逻辑。

### 1.1 技术栈与运行环境

| 项 | 说明 |
|---|---|
| 框架 | Spring Boot 4.1.0（内嵌 Tomcat） |
| 语言 | Java 21 |
| 数据访问 | Spring `JdbcTemplate` |
| 服务端口 | **8080** |
| 数据库 | MySQL `bigdata`（本机 `localhost:3306`，账号 `root/123456`） |
| 前端 | Vue3 + Vite + ECharts + axios（开发端口 5173，经 Vite 代理转发 `/api`） |
| AI 模块 | 纯 JDK 实现，调用 DeepSeek `deepseek-v4-pro`（无第三方 JSON 库依赖） |

### 1.2 基础 URL

```
http://localhost:8080/api
```

- 后端接口统一以 `/api` 为前缀。
- 前端开发环境通过 Vite 代理：`/api` → `http://localhost:8080`。
- 生产环境可将前端 `baseURL` 改为完整后端地址，或反向代理到 `/api`。

## 2. 通用约定

### 2.1 响应信封（统一格式）

所有接口返回统一 JSON 信封 `{code, msg, data}`：

```json
{
  "code": 0,
  "msg": "success",
  "data": [ /* 业务数据 */ ]
}
```

| 字段 | 类型 | 说明 |
|---|---|---|
| `code` | int | `0` 表示成功，非 `0` 表示失败 |
| `msg` | String | 成功为 `"success"`；失败为错误描述 |
| `data` | Object/Array | 业务数据负载（结构因接口而异） |

> 数据访问层使用 Lombok `@Data`，字段为 `code / msg / data`。

### 2.2 状态码约定

| code | 含义 | 触发场景 |
|---|---|---|
| `0` | 成功 | 正常返回 `data` |
| `1` | 业务失败 | 指标不存在 / 查询异常 / 消息为空 / 模型调用失败等，详见 `msg` |

> HTTP 状态码始终为 `200`（即便业务失败），业务成败通过 `code` 判断。前端约定：**`code === 0` 视为成功**。

### 2.3 字符与编码

- 请求/响应均为 `UTF-8`，`Content-Type: application/json`。
- 跨域（CORS）已在 `CorsConfig` 中放开 `http://localhost:5173` 与 `http://127.0.0.1:5173`，允许任意 Header 与 Method。

## 4. 接口详述

### 4.1 GET /api/metrics

返回全部指标的"元信息"列表。前端在初始化时调用一次，据此决定每个图表如何渲染（图表类型、维度字段、数值字段、坐标轴顺序、单位）。

**请求参数**：无

**响应 `data`**：`MetricMeta` 数组，元素字段如下：

| 字段 | 类型 | 说明 |
|---|---|---|
| `board` | String | 板块（URL 第一段） |
| `metric` | String | 指标（URL 第二段） |
| `title` | String | 中文展示标题 |
| `chartType` | String | 图表类型：`barH` / `pie` / `line` / `heatmap` / `wordcloud` / `stackedBar` |
| `xField` | String | 分类/维度字段名 |
| `yField` | String | 热力图 y 轴字段名（仅 heatmap 用） |
| `valueField` | String | 数值度量字段名 |
| `seriesField` | String | 堆叠柱图系列字段（仅 stackedBar 用） |
| `categoryField` | String | 堆叠柱图分组（x 轴）字段（仅 stackedBar 用） |
| `topN` | Integer | 堆叠柱图取前 N 个分组（仅 stackedBar 用） |
| `xOrder` | List<String> | x 轴分类顺序提示（用于坐标轴排序） |
| `yOrder` | List<String> | 热力图 y 轴顺序提示 |
| `unit` | String | 单位（`元` / `个` / `%` / `次`） |

> 注：`MetricDef` 内部还含 `tableName / orderBy / limit / groupBy` 等字段，但这些属于后端内部映射，**不对外暴露**，因此不在 `MetricMeta` 中。

**响应示例（节选）**：

```json
{
  "code": 0,
  "msg": "success",
  "data": [
    {
      "board": "talent",
      "metric": "salary",
      "title": "学历 × 经验 → 平均薪资",
      "chartType": "heatmap",
      "xField": "exp_level",
      "yField": "edu_level",
      "valueField": "avg_salary",
      "seriesField": null,
      "categoryField": null,
      "topN": null,
      "xOrder": ["经验不限","无经验","1年以下","1-3年","3-5年","5-10年","10年以上"],
      "yOrder": ["MBA/EMBA","EMBA","博士","硕士","本科","大专","中专/中技","高中","初中及以下","学历不限"],
      "unit": "元"
    }
  ]
}
```

### 4.2 GET /api/{board}/{metric}

根据板块与指标，返回对应 ADS 结果表的数据行（`List<Map>`）。前端 `MetricChart` 组件读取后绑定到对应 ECharts 图表。

**路径参数**：

| 参数 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `board` | String | 是 | 板块名（如 `talent` / `geo` / `industry` / `time` / `text`） |
| `metric` | String | 是 | 指标名（如 `salary` / `demand` / `trend`） |

**查询逻辑（后端内部）**：
1. 通过 `MetricRegistry.find(board, metric)` 查找指标定义；
2. 若未找到，返回 `code=1, msg="指标不存在: {board}/{metric}"`；
3. 组装 SQL：
   - 若配置了 `groupBy`：`SELECT \`groupBy\`, SUM(\`valueField\`) AS \`valueField\` FROM \`table\` GROUP BY \`groupBy\``；
   - 否则：`SELECT * FROM \`table\``；
   - 追加 `ORDER BY {orderBy}`（若配置）；
   - 若 `limit > 0`：追加 `LIMIT {limit}`；**`limit <= 0` 表示返回全量**（用于 `geo/demand` 等大屏需要全量数据的指标）；
4. `jdbc.queryForList(...)` 返回行；异常时返回 `code=1, msg="查询失败: ..."`。

**响应 `data`**：对象数组，每个对象的键为表中列名。

**请求示例**：

```
GET /api/talent/edu
GET /api/geo/demand
GET /api/time/trend
```

**响应示例（/api/talent/edu）**：

```json
{
  "code": 0,
  "msg": "success",
  "data": [
    {"education": "本科", "job_count": 59193},
    {"education": "大专及以下", "job_count": 26834},
    {"education": "研究生及以上", "job_count": 4461},
    {"education": "高中及以下", "job_count": 439}
  ]
}
```

> 实际字段名以各 ADS 表结构为准（如学历表为 `education`/`job_count`，薪资表为 `city`/`avg_salary` 等）。前端按元信息里的 `xField / valueField` 取值，不写死列名。

**错误示例**：

```json
{ "code": 1, "msg": "指标不存在: talent/unknown", "data": null }
```

### 4.4 POST /api/ai/chat

与大模型对话。系统提示词中**写死了平台核心真实统计数据**，模型仅依据这些数据作答，不编造。

**请求头**：`Content-Type: application/json`

**请求体**：

```json
{
  "message": "北京的平均薪资大概多少？",
  "history": [
    { "role": "user", "content": "哪个城市岗位最多？" },
    { "role": "assistant", "content": "北京岗位最多，共约 16650 个。" }
  ]
}
```

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `message` | String | 是 | 当前用户问题；为空时返回 `code=1, msg="消息不能为空"` |
| `history` | Array | 否 | 历史消息列表，每项 `{role, content}`；仅 `user` / `assistant` 角色且 `content` 非空者被采用，用于多轮上下文 |

**处理流程（后端）**：
1. 合法校验 `message`；
2. 用 `StringBuilder` 手工拼装 `messages` JSON 数组：首条为 `system`（含写死的平台数据 + 回答规则），随后追加 `history`，最后追加当前 `user` 消息；
3. 组装请求体：`model=deepseek-v4-pro`、`temperature=0.3`、`max_tokens=1500`、`stream=false`；
4. 通过 JDK `HttpClient` POST 到 `https://api.deepseek.com/v1/chat/completions`，携带 `Authorization: Bearer {API_KEY}`；
5. 超时 90s；HTTP 非 200 返回 `code=1`；
6. 用自写 `extractContent()` 从响应 JSON 字符串中提取 `choices[0].message.content`（推理模型带 `reasoning_content`，此处只取 `content`，无需 Jackson）；
7. 返回 `{ reply: "<模型回复文本>" }`。

**响应 `data`**：

```json
{ "reply": "根据平台数据，北京招聘岗位数位居全国第一，约 16650 个，平均月薪约 11538 元/月（本科口径）。数据参考维度：招聘需求 TOP 城市、学历平均薪资。" }
```

**错误示例**：

```json
{ "code": 1, "msg": "模型服务返回错误（HTTP 401）", "data": null }
```

**配置说明（写死于 `AiController`）**：

| 常量 | 值 |
|---|---|
| `MIMO_URL` | `https://api.deepseek.com/v1/chat/completions` |
| `MIMO_API_KEY` | 通过环境变量 `DEEPSEEK_API_KEY` 注入（占位符 `YOUR_DEEPSEEK_API_KEY` 兜底，请勿将真实 Key 提交到公开仓库） |
| `MIMO_MODEL` | `deepseek-v4-pro`（**注意大小写敏感**，官方仅认全小写） |

> 纯 JDK 实现：用 `jsonEscape()` 手工处理 JSON 转义、用 `extractContent()` 字符串定位提取字段，**不引入 Jackson 等 JSON 库**，因此 `pom.xml` 无需额外 JSON 依赖，规避了受控环境无法下载 Maven 中央仓库依赖的问题。

## 6. 前端 API 封装

文件：`frontend/src/api/metric.js`

```js
import axios from 'axios'

const http = axios.create({
  baseURL: '/api',      // 开发期经 Vite 代理转发到后端 8080
  timeout: 20000
})

export async function getMetrics() {
  const { data } = await http.get('/metrics')
  return data.data || []            // 返回 MetricMeta[]（已取信封内 data）
}

export async function getMetricData(board, metric) {
  const { data } = await http.get(`/${board}/${metric}`)
  return data.data || []            // 返回数据行数组（已取信封内 data）
}
```

> 前端对**每个接口都判断 `code === 0` 才算成功**，统一从 `data.data` 取业务负载。AI 对话单独在 `ChatWidget.vue` 中调用 `POST /api/ai/chat`，并设 90s 超时（推理模型较慢）。

## 7. AI 问答模块说明

- **接入模型**：DeepSeek `deepseek-v4-pro`（官方地址，无需中转）。
- **数据源**：系统提示词内写死平台真实统计（学历分布、经验、薪资、TOP 城市/省份、行业、月度趋势、技能词云、福利提及率、热门企业/职位），数值来自数仓分析结果，力求准确、不编造。
- **交互**：前端右下角可拖拽浮窗 `ChatWidget.vue`，仅支持 **ESC 关闭**（按需求去掉了 ✕ 按钮）。
- **多轮**：前端维护 `history` 并随每次请求回传，实现上下文连贯。
- **安全**：`message` 为空会被后端拒绝；模型被约束"只依据真实数据作答、不讨论敏感话题"。

## 9. 附录：字段类型与图表映射速查

| chartType | 用途 | 关键字段 |
|---|---|---|
| `barH` | 横向柱状（排名类） | `xField`（类目）、`valueField`（数值） |
| `pie` | 饼图（占比类） | `xField`（类目）、`valueField`（数值） |
| `line` | 折线（趋势类） | `xField`（时间/序）、`valueField`（数值） |
| `heatmap` | 热力矩阵 | `xField`（x 轴）、`yField`（y 轴）、`valueField`（色值） |
| `stackedBar` | 堆叠柱图 | `seriesField`（系列）、`categoryField`（x 分组）、`topN` |
| `wordcloud` | 词云 | `xField`（词）、`valueField`（权重） |

> 新增指标：只需在 `MetricRegistry.java` 的 `List.of(...)` 中追加一条 `MetricDef`，无需新建 Controller 或前端代码；前端会根据新元信息自动渲染。
